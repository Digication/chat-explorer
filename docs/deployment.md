# Deployment

This document describes how the Chat Explorer app is deployed to production and the decisions behind that setup. If you're new to the project and need to deploy, update, or troubleshoot the production environment, start here.

## TL;DR

- **Hosted on:** [Railway](https://railway.com) (Pro plan)
- **Current URL:** https://chat-explorer.up.railway.app
- **Target custom domain:** `chat-explorer.digication.com` (planned — see [Custom Domain Setup](#custom-domain-setup))
- **Database:** Railway-managed PostgreSQL (same project)
- **Deploys:** Automatic on push to `main` branch of [Digication/chat-explorer](https://github.com/Digication/chat-explorer)
- **Build:** Railway runs `pnpm build` (Vite for client, `tsc` for server), then `pnpm start` (`node dist/server/index.js`)
- **Migrations:** TypeORM migrations run automatically on server startup in production

## Architecture

```
GitHub (main branch)
        │
        │ auto-deploy on push
        ▼
┌───────────────────────────┐    ┌────────────────────┐
│  Railway: chat-explorer   │◄──►│ Railway: Postgres  │
│  (Express + React bundle) │    │ (managed DB)       │
└───────────────────────────┘    └────────────────────┘
        ▲
        │ HTTPS
        ▼
      Users
```

In production, Express serves both the API **and** the bundled React frontend from the same origin. This is different from dev, where Vite runs on its own port (5173 / `chat-explorer.localhost`) and proxies API calls to Express on `localhost:4000`.

## Environment Variables

Set in the Railway dashboard under **chat-explorer service → Variables**.

### Required

| Variable | Purpose | Source |
|---|---|---|
| `NODE_ENV` | Enables production code paths (static serving, migration runner, etc.) | Set to `production` |
| `PORT` | Port Express listens on | Set to `3000` (Railway expects this) |
| `DATABASE_URL` | Postgres connection string | Reference: `${{Postgres.DATABASE_URL}}` (links to Railway's managed Postgres) |
| `BETTER_AUTH_SECRET` | Signing key for auth sessions | Generate with `openssl rand -hex 32` |
| `BETTER_AUTH_URL` | Canonical URL of the deployed app | Currently `https://chat-explorer.up.railway.app` |
| `GOOGLE_CLIENT_ID` | Google OAuth client ID | Google Cloud Console → APIs & Services → Credentials |
| `GOOGLE_CLIENT_SECRET` | Google OAuth client secret | Same as above |
| `OPENAI_API_KEY` | LLM API key | platform.openai.com |
| `ANTHROPIC_API_KEY` | LLM API key | console.anthropic.com |
| `GOOGLE_AI_API_KEY` | LLM API key (Gemini) | Google AI Studio |

### Optional

| Variable | Purpose |
|---|---|
| `ALLOWED_ORIGINS` | Comma-separated list of extra origins for CORS. Useful if you're transitioning between URLs (e.g. during the custom domain rollout). The CORS allowlist already includes `BETTER_AUTH_URL` automatically. |

### Auto-injected by Railway (don't set manually)

Railway provides `RAILWAY_PRIVATE_DOMAIN`, `RAILWAY_PROJECT_NAME`, `RAILWAY_ENVIRONMENT_NAME`, `RAILWAY_SERVICE_NAME`, `RAILWAY_PROJECT_ID`, `RAILWAY_ENVIRONMENT_ID`, `RAILWAY_SERVICE_ID`. The app doesn't currently read any of these, but they're available if needed.

## Database Migrations

**Important:** In production, TypeORM's `synchronize` is **disabled**. Schema changes must go through explicit migration files.

### How it works

1. Migration files live in `src/server/migrations/`.
2. Each is imported and registered in `src/server/data-source.ts` under the `migrations:` array.
3. On production server startup (`src/server/index.ts`), the app calls `AppDataSource.runMigrations()` which applies any pending migrations before the server starts accepting requests.
4. TypeORM tracks which migrations have run via a `migrations` table it creates automatically.

### Current migrations

| File | What it creates |
|---|---|
| `1775574106489-Initial.ts` | All TypeORM entity tables (Institution, User, Course, Assignment, Thread, Student, Comment, ToriTag, CommentToriTag, StudentConsent, CourseAccess, UploadLog, ChatSession, ChatMessage, UserState) plus indexes, foreign keys, and enum types |
| `1775574200000-AddBetterAuthTables.ts` | Better Auth's `session`, `account`, and `verification` tables (managed outside TypeORM) |

### Generating new migrations

The standard TypeORM workflow (`pnpm migration:generate`) compares your entities against a live database and emits a diff. In this project:

- **In dev:** The Docker dev database has `synchronize: true`, which means its schema is always in sync with the entities. Running `migration:generate` against it will produce an empty migration. Use a scratch database or the `scripts/generate-initial-migration.mjs` approach instead.
- **The existing initial migration** was generated by pointing `scripts/generate-initial-migration.mjs` at the (then-empty) Railway Postgres public URL. This script uses TypeORM's `driver.createSchemaBuilder().log()` to dump the CREATE TABLE SQL and format it as a migration class.

For changes to **Better Auth's schema** (e.g. if you add plugins or upgrade the library), use Better Auth's own CLI:

```bash
# Generate SQL for the current better-auth config
DATABASE_URL=<any-postgres-url> \
BETTER_AUTH_SECRET=stub GOOGLE_CLIENT_ID=stub GOOGLE_CLIENT_SECRET=stub \
npx @better-auth/cli@latest generate \
  --config dist/server/auth.js \
  --output /tmp/better-auth-schema.sql -y
```

Then wrap the diff in a new TypeORM migration file and register it in `data-source.ts`.

## How the Frontend is Served in Production

In `src/server/index.ts`, when `NODE_ENV === "production"`:

1. `express.static(dist/client)` serves the bundled assets (JS, CSS, images).
2. A catch-all GET route (`/^\/(?!api|graphql|auth).*/`) sends `dist/client/index.html` for any non-API route, so client-side routing (React Router) works on hard refresh / direct links.

This means you don't need a separate static host (S3, CDN, etc.) — Express handles everything. Good enough for current traffic; if load becomes an issue, the client bundle could be moved to a CDN later.

## API Base URL Handling

Because the dev and production setups have different cross-origin behavior, the frontend uses a shared helper in `src/lib/api-base.ts`:

```typescript
export const API_BASE = import.meta.env.PROD ? "" : "http://localhost:4000";
```

- **In dev:** Frontend on `chat-explorer.localhost` talks to API on `localhost:4000`. Hardcoded because auth cookies are scoped to that host.
- **In production:** Same origin, so `API_BASE` is an empty string and requests resolve to the current host.

All four API consumers use this helper:
- `src/lib/auth-client.ts` (Better Auth client)
- `src/lib/apollo-client.ts` (GraphQL)
- `src/components/upload/CsvUploadCard.tsx` (file uploads)
- `src/pages/LoginPage.tsx` (Google sign-in redirect)

**If you add a new API call from the frontend, use `API_BASE` — don't hardcode URLs.**

## Google OAuth Authorized Redirects

The Google OAuth client (in the Digication Google Cloud project) must list every URL the app runs on as an **Authorized redirect URI**. Currently:

- `https://chat-explorer.localhost/api/auth/callback/google` (dev)
- `https://chat-explorer.up.railway.app/api/auth/callback/google` (production)
- *(Planned)* `https://chat-explorer.digication.com/api/auth/callback/google`

Adding a new environment? Add the corresponding redirect URI in [Google Cloud Console → Credentials](https://console.cloud.google.com/apis/credentials) before deploying, or sign-in will silently fail.

## Deploying Changes

Normal workflow:

1. Make changes on a branch, test locally via Docker
2. Merge to `main` (or push directly if you're the only one working on it)
3. Railway detects the push and auto-deploys
4. Watch the deploy in the Railway dashboard → **chat-explorer** → **Deployments**
5. Tail logs with `railway logs --lines 50` (requires `railway link` to be set up locally)

**If the deploy fails**, the previous version keeps running — Railway doesn't replace the live instance until the new one passes its health check.

## Custom Domain Setup

Not yet done for production, but the plan:

1. In Railway: **chat-explorer service → Settings → Networking → + Custom Domain** → enter `chat-explorer.digication.com`. Railway provides a CNAME target.
2. Add a CNAME record in Digication's DNS provider: `chat-explorer` → the value Railway gave.
3. Wait for DNS propagation (5–30 min). Railway auto-provisions an SSL cert via Let's Encrypt once it sees the record.
4. Update `BETTER_AUTH_URL` in Railway Variables to `https://chat-explorer.digication.com`.
5. Add the new callback URL to Google OAuth Authorized redirect URIs.
6. (Optional) Set `ALLOWED_ORIGINS=https://chat-explorer.up.railway.app` during the transition to keep the old URL working.

The old `*.up.railway.app` URL continues to work even after a custom domain is added, so there's no hard cutover.

## Staging Environment

A staging server at `https://chat-explorer.digistage.me` for testing new features (especially the outcomes/evidence phases) before they hit production.

### Architecture

```
GitHub (staging branch)
        │
        │ auto-deploy on push
        ▼
┌──────────────────────────────┐    ┌─────────────────────────┐
│  Railway: chat-explorer-stg  │◄──►│ Railway: Postgres (stg) │
│  (Express + React bundle)    │    │ (managed DB)            │
└──────────────────────────────┘    └─────────────────────────┘
        ▲
        │ HTTPS via chat-explorer.digistage.me
        ▼
      Testers
```

Same setup as production — Express serves the bundled React app and the API from one process. Its own database, its own env vars, its own deploy trigger (the `staging` branch).

### Setup Steps

#### 1. Create Railway service

In the existing Railway project (keeps billing together):

1. **+ New Service → GitHub Repo** → select `Digication/chat-explorer`
2. **Settings → Source → Branch** → set to `staging`
3. **Settings → Build** → Build command: `pnpm build`, Start command: `pnpm start`
4. Rename the service to `chat-explorer-staging` in Railway

#### 2. Add a staging database

1. **+ New Service → Database → PostgreSQL**
2. Rename it to `Postgres (staging)` to distinguish from production
3. In the `chat-explorer-staging` service, reference the staging database:
   - Set `DATABASE_URL` to `${{Postgres (staging).DATABASE_URL}}`

#### 3. Set environment variables

In **chat-explorer-staging → Variables**:

| Variable | Value |
|----------|-------|
| `NODE_ENV` | `production` |
| `PORT` | `3000` |
| `DATABASE_URL` | `${{Postgres (staging).DATABASE_URL}}` |
| `BETTER_AUTH_SECRET` | Generate new: `openssl rand -hex 32` (different from prod!) |
| `BETTER_AUTH_URL` | `https://chat-explorer.digistage.me` |
| `GOOGLE_CLIENT_ID` | Same as production (same Google OAuth app) |
| `GOOGLE_CLIENT_SECRET` | Same as production |
| `GOOGLE_AI_API_KEY` | Same as production (or a separate key with lower quota) |
| `OPENAI_API_KEY` | Same as production |
| `ANTHROPIC_API_KEY` | Same as production |
| `SENDGRID_API_KEY` | Same as production (or omit — magic links will log to console) |
| `SENDGRID_FROM_EMAIL` | `noreply@digication.com` |
| `BOOTSTRAP_ADMIN_EMAIL` | Your email (to get notified of sign-in attempts) |

**Important:** `BETTER_AUTH_SECRET` must be different from production. If they share the same secret, a session cookie from one environment could authenticate in the other.

#### 4. Custom domain

1. In Railway: **chat-explorer-staging → Settings → Networking → + Custom Domain** → enter `chat-explorer.digistage.me`
2. Railway gives you a CNAME target (something like `abc123.up.railway.app`)
3. In your DNS provider for `digistage.me`: add a CNAME record:
   - Name: `chat-explorer`
   - Target: the value Railway provided
4. Wait for DNS propagation (5–30 min). Railway auto-provisions SSL via Let's Encrypt.

#### 5. Google OAuth redirect URI

In [Google Cloud Console → Credentials](https://console.cloud.google.com/apis/credentials), add to the existing OAuth client's **Authorized redirect URIs**:

```
https://chat-explorer.digistage.me/api/auth/callback/google
```

This is required for Google sign-in to work on the staging domain. Without it, sign-in silently fails.

#### 6. Create the staging branch

```bash
git checkout main
git checkout -b staging
git push -u origin staging
```

Railway auto-deploys when this branch is pushed. To get new code to staging:

```bash
# Merge main into staging (or cherry-pick specific commits)
git checkout staging
git merge main
git push
```

Or merge feature branches directly into `staging` for testing before they go to `main`.

#### 7. Seed staging data

The staging database starts empty. On first deploy, migrations run automatically and TORI tags are seeded. To test with realistic data:

- Upload a CSV via the staging app's upload page, OR
- Copy a subset of production data using `pg_dump`/`pg_restore` (selective — don't copy user sessions or auth tables)

### Workflow

| Action | Production | Staging |
|--------|-----------|---------|
| Deploy trigger | Push to `main` | Push to `staging` |
| URL | `chat-explorer.up.railway.app` | `chat-explorer.digistage.me` |
| Database | Shared with real users | Isolated, safe to reset |
| Auth | Same Google OAuth app | Same Google OAuth app, different session secret |
| Testing new features | Never | Always test here first |

### Resetting the staging database

If the staging database gets into a bad state (e.g., a failed migration during development), you can reset it:

1. In Railway dashboard: open the staging Postgres service
2. Delete the service and create a new one
3. Update the `DATABASE_URL` reference in `chat-explorer-staging`
4. Redeploy — migrations will run fresh on the empty database

### Cost

Railway Pro plan includes $20/month in credits. The staging service and its database share that allowance. A low-traffic staging environment typically costs $2–5/month (mostly the Postgres instance). If it goes unused, you can scale the staging service to zero in Railway settings.

## Key Decisions and History

### Why Railway (not Vercel / Render / AWS)?

- Single platform for app + database + env var management
- Automatic GitHub integration with branch-based previews
- Free Let's Encrypt SSL for custom domains
- Managed Postgres in the same dashboard
- Pro plan ($20/mo) gets priority during incidents — Hobby deploys were paused during an outage while we were setting this up, which is how we found out Pro is worth it for production workloads

### Why Pro plan over Hobby?

During the initial setup (April 6–8, 2026), Railway had a deployment incident where Hobby-tier deploys were paused to protect Pro users. CLI writes were returning "Unauthorized" for Hobby users. Upgrading to Pro resolved the immediate incident-related failures and ensures the app isn't deprioritized during future incidents. Pro also gives $20 of resource credits included, so for a small app there's usually no overage.

### Why explicit migrations instead of `synchronize: true` in production?

TypeORM's `synchronize` option auto-creates tables from entity definitions. It's convenient in dev but dangerous in production — it can silently drop columns when entity definitions change, leading to data loss. Explicit migrations give you a reviewable diff and a rollback path. The initial migration was generated via a helper script (`scripts/generate-initial-migration.mjs`) because the project didn't have any migrations when we deployed.

### Why Express serves the React bundle (instead of a separate static host)?

Simpler. One deployable, one env var namespace, one SSL cert, one set of logs. If the app grows enough that serving static assets becomes a bottleneck, we can move `dist/client` to a CDN (Cloudflare, Bunny, etc.) without touching anything else.

### Why API_BASE helper instead of a build-time env var?

`import.meta.env.PROD` is built into Vite and requires zero configuration. A build-time env var would work too, but it's one more thing to remember to set in Railway and keep in sync across environments.

## Troubleshooting

### "Cannot find module '/app/dist/server/index.js'"

The build step didn't produce the server output. Check that `tsconfig.server.json` doesn't have `noEmit: true` and that the build command in `package.json` runs the server compile (not just Vite).

### "relation 'X' does not exist"

A migration didn't run or is missing. Check:
1. `railway logs` — did you see "Running pending migrations..." followed by "Migrations complete."?
2. Is the missing table from TypeORM entities or from Better Auth?
   - TypeORM: should be in the initial migration. Regenerate via `scripts/generate-initial-migration.mjs` if you've added entities.
   - Better Auth: needs its own migration (see [Generating new migrations](#generating-new-migrations)).

### Login redirects to `localhost`

The frontend has a hardcoded `http://localhost:4000` somewhere. It should use `API_BASE` from `src/lib/api-base.ts`. This happened during initial deploy — search the client code for `localhost:4000` and replace with `${API_BASE}`.

### Sign-in fails with "Unexpected end of JSON input"

The Better Auth endpoint (`/api/auth/sign-in/social`) is returning a 500 with no body. Check logs — most likely a missing auth table in the database (see above).

### CORS errors in the browser console

The origin making the request isn't in the allowlist in `src/server/index.ts`. Either:
- Set `BETTER_AUTH_URL` in Railway Variables to match the origin you're using, OR
- Add additional origins via the `ALLOWED_ORIGINS` env var (comma-separated).

### Deploy succeeds but app shows "Cannot GET /"

Express isn't serving the static files. Check that `NODE_ENV=production` is set in Railway Variables — the static serving block in `index.ts` is guarded behind that check.
